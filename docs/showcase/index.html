Showcase

powerglide was built to run unattended, and it would be hollow to ship it without having actually taken it around the track. This page documents real triage sessions — powerglide driving itself with local Qwen3.5 models served through igllama, with no cloud API required. The goal: understand what small models can do inside the Ralph Loop, where they break down, and how to get the most out of sub-4B parameter inference on local hardware.

Setup: igllama + powerglide

igllama is a Zig-based Ollama alternative — a single static binary that serves GGUF models over an OpenAI-compatible REST API. Combined with powerglide’s openai_compat provider support, it forms a fully local, zero-dependency agentic stack.

hardware:  server-class x86_64 (CPU inference, no GPU)
models:    Qwen3.5-0.8B-Q8_0.gguf  ← igllama on :8090
           Qwen3.5-4B-Q8_0.gguf   ← igllama on :8091
framework: powerglide v0.2.1, igllama v0.3.7

Starting igllama

# Serve 0.8B on port 8090 — suppresses chain-of-thought blocks for faster responses
igllama api Qwen3.5-0.8B-Q8_0.gguf --port 8090 --no-think &

# Serve 4B on port 8091
igllama api Qwen3.5-4B-Q8_0.gguf --port 8091 --no-think &

Case Study 3 — Tool Calling Triage

This is the honest assessment: small models do not reliably emit structured tool call JSON. The OpenAI /v1/chat/completions tools parameter expects the model to return a tool_calls array with precise JSON — but both the 0.8B and 4B Qwen3.5 models tend to write markdown-fenced code blocks instead when a bash tool is offered.

Raw API test (4B):

$ curl http://127.0.0.1:8091/v1/chat/completions \
    -d '{"model":"Qwen3.5-4B-Q8_0.gguf",
         "messages":[{"role":"user","content":"list files in src/"}],
         "tools":[{"type":"function","function":{"name":"bash","description":"run bash","parameters":{"type":"object","properties":{"command":{"type":"string"}},"required":["command"]}}}],
         "tool_choice":"auto"}'

Response (truncated):

{
  "choices": [{
    "message": {
      "content": "To list files in src/, I'll use the bash tool:\n\n```bash\nls src/\n```",
      "tool_calls": null
    }
  }]
}

The model knows what to do but doesn’t produce the structured tool_calls payload. It writes prose + fenced code instead.

Why this happens: Reliable tool call formatting requires the model to have been trained on the exact JSON schema format for function calls. At sub-4B scale, the Qwen3.5 models are generalist; the tool-call training signal competes with many other behaviours and doesn’t always win.

The fix — JSON mode: powerglide’s OpenAIClient now supports a json_mode flag that sets response_format: {"type": "json_object"} in the request. igllama honours this flag and engages constrained generation, forcing the output to be valid JSON. Used with a system prompt that specifies the tool call schema explicitly, this reliably elicits structured output from the 4B model — turning prose + fenced code into a parseable tool invocation.

# Constrained JSON generation in igllama
# Forces {"name": "bash", "arguments": {"command": "ls src/"}} instead of prose

Recommended workflow for local models: Use the local/local4b agents for exploration, summarization, and Q&A tasks where the loop’s THINKING and OBSERVING states handle the interaction — not for multi-step code writing that requires reliable tool invocations. For agentic code modification, the hephaestus (Claude Opus) agent is the right choice.

This is meaningful for dogfooding: you can see at a glance how many loop iterations a task required, how long it took wall-clock, and whether it ended cleanly (<POWERGLIDE_DONE>) or with an error (<POWERGLIDE_ERROR>). It also makes the session summary scriptable — grep POWERGLIDE_DONE in CI is now the reliable completion check.

Two engineering lessons learned before this worked:

  1. json_object mode produces trailing garbage. igllama sometimes appends an extra } or a stray newline after the closing brace. The solution is a brace-counter extractor, not json.loads(raw) directly.
  2. igllama throttles under load. The 4B model runs slow (20–110s/call on CPU); back-to-back requests often hit 400 No messages provided — not a real error, just the server briefly overloaded. A 3-attempt retry with 5s backoff absorbed all of them cleanly.

Engineering requirements for reliable local inference

From the trial, four things are essential for reliable multi-turn tool use with a 4B model:

  1. JSON-mode (response_format: {"type":"json_object"}) — without it the model writes prose with fenced code blocks instead of parseable JSON
  2. Brace-balanced JSON extractor — json.loads() fails on the raw response roughly 30% of the time; a tolerance-aware extractor recovers silently
  3. Retry with backoff on 400 — igllama returns spurious 400s under load; 3 retries × 5s absorbs them
  4. Context window capping — keeping only the last 3 tool/result pairs prevents the request body from exceeding igllama’s context limit (which also produces 400s)

Dogfooding Verdict

The Ralph Loop is validated at every scale tested. The explicit state machine keeps the agent on track whether the model is 4B running locally or Claude Opus running in the cloud — it drives the model through states, it doesn’t rely on the model to self-sequence.

For multi-turn coding tasks with a local 4B model, the requirements are concrete:

  • json_mode to force structured output
  • A brace-tolerant JSON parser to handle igllama’s trailing-byte edge cases
  • Retry logic for server throttling
  • Context trimming to stay inside the inference window

With those four things in place, Qwen3.5-4B at Q8_0 quantization completes multi-step read → write → verify workflows reliably, self-corrects from tool errors, and maintains coherent context across 4+ turns. That is the full agentic coding loop — just slower, and free.
