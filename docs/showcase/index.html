Showcase

powerglide was built to run unattended, and it would be hollow to ship it without having actually taken it around the track. This page documents real triage sessions — powerglide driving itself with local Qwen3.5 models served through igllama, with no cloud API required. The goal: understand what small models can do inside the Ralph Loop, where they break down, and how to get the most out of sub-4B parameter inference on local hardware.

Setup: igllama + powerglide

igllama is a Zig-based Ollama alternative — a single static binary that serves GGUF models over an OpenAI-compatible REST API. Combined with powerglide’s openai_compat provider support, it forms a fully local, zero-dependency agentic stack.

hardware:  server-class x86_64 (CPU inference, no GPU)
models:    Qwen3.5-0.8B-Q8_0.gguf  ← igllama on :8090
           Qwen3.5-4B-Q8_0.gguf   ← igllama on :8091
framework: powerglide v0.2.1, igllama v0.3.7

Starting igllama

# Serve 0.8B on port 8090 — suppresses chain-of-thought blocks for faster responses
igllama api Qwen3.5-0.8B-Q8_0.gguf --port 8090 --no-think &

# Serve 4B on port 8091
igllama api Qwen3.5-4B-Q8_0.gguf --port 8091 --no-think &

Case Study 3 — Tool Calling Triage

This is the honest assessment: small models do not reliably emit structured tool call JSON. The OpenAI /v1/chat/completions tools parameter expects the model to return a tool_calls array with precise JSON — but both the 0.8B and 4B Qwen3.5 models tend to write markdown-fenced code blocks instead when a bash tool is offered.

Raw API test (4B):

$ curl http://127.0.0.1:8091/v1/chat/completions \
    -d '{"model":"Qwen3.5-4B-Q8_0.gguf",
         "messages":[{"role":"user","content":"list files in src/"}],
         "tools":[{"type":"function","function":{"name":"bash","description":"run bash","parameters":{"type":"object","properties":{"command":{"type":"string"}},"required":["command"]}}}],
         "tool_choice":"auto"}'

Response (truncated):

{
  "choices": [{
    "message": {
      "content": "To list files in src/, I'll use the bash tool:\n\n```bash\nls src/\n```",
      "tool_calls": null
    }
  }]
}

The model knows what to do but doesn’t produce the structured tool_calls payload. It writes prose + fenced code instead.

Why this happens: Reliable tool call formatting requires the model to have been trained on the exact JSON schema format for function calls. At sub-4B scale, the Qwen3.5 models are generalist; the tool-call training signal competes with many other behaviours and doesn’t always win.

The fix — JSON mode: powerglide’s OpenAIClient now supports a json_mode flag that sets response_format: {"type": "json_object"} in the request. igllama honours this flag and engages constrained generation, forcing the output to be valid JSON. Used with a system prompt that specifies the tool call schema explicitly, this reliably elicits structured output from the 4B model — turning prose + fenced code into a parseable tool invocation.

# Constrained JSON generation in igllama
# Forces {"name": "bash", "arguments": {"command": "ls src/"}} instead of prose

Recommended workflow for local models: Use the local/local4b agents for exploration, summarization, and Q&A tasks where the loop’s THINKING and OBSERVING states handle the interaction — not for multi-step code writing that requires reliable tool invocations. For agentic code modification, the hephaestus (Claude Opus) agent is the right choice.

Case Study 4 — Session Summary Output (v0.2.1)

One of the v0.2.1 improvements was making powerglide run emit a structured session summary on completion. Previously, the command finished silently — you’d see the Ralph loop log lines and nothing else. Now every session ends with:

─────────────────────────────────────────
  Session complete  [done]
  Steps:    23
  Elapsed:  9.0s
  Agent:    local  (Qwen3.5-0.8B-Q8_0.gguf)
  Signal:   <POWERGLIDE_DONE>
─────────────────────────────────────────

This is meaningful for dogfooding: you can see at a glance how many loop iterations a task required, how long it took wall-clock, and whether it ended cleanly (<POWERGLIDE_DONE>) or with an error (<POWERGLIDE_ERROR>). It also makes the session summary scriptable — grep POWERGLIDE_DONE in CI is now the reliable completion check.

Case Study 5 — Full Multi-Turn Agentic Trial (4B, 6 tasks)

This is the most thorough triage: a purpose-built agentic loop (pg_trial.py) drives Qwen3.5-4B through six real coding tasks using JSON-mode tool dispatch, executes every tool call for real, and feeds results back across multiple turns. No scaffolding, no simulated answers. Every bash command ran. Every file that was written was read back.

Task results

     grep found: const VERSION = "0.2.2"

T2 — Bash: count LoopState variants  ✗  1 turn   370s  [igllama timeout — no --threads opt]

T3 — Write: Zig clamp function       ✗  1 turn   370s  [igllama timeout — no --threads opt]

T4 — Read + Write: summarise swarm   ✓  4 turns  203s
     head → write → cat → done

T5 — Grep: TODO comments in src/     ✓  3 turns   49s
     Confirmed: zero TODO comments across all 22 .zig files

T6 — Multi-file struct names         ✓  3 turns   67s
     Self-corrected: "bash grep" → "grep" after exit 126

T2 and T3 timed out because the original igllama instance had no --max-tokens cap and no thread tuning. Root cause: without --threads 4, igllama uses all 16 cores at once. More cores competing for the same memory channels means lower per-token throughput, not higher — the model hits the 120s urllib timeout on every attempt (3 retries × 120s = 360s).

Case Study 6 — Extended Trial After igllama Optimization

With T2 and T3 blocked by timeout, the natural question was: is this a model capacity problem or a server configuration problem? The answer is the latter. Restarting igllama with --threads 4 --threads-batch 16 --ctx-size 2048 --mlock dropped the per-call latency from 120s+ to 10–65s, and T3 immediately passed in the next run.

This case study is the extended version: 13 tasks (pg_trial_v2.py) across three model weight classes — 0.5B coder, 0.8B generalist, 4B generalist — all on CPU, all with optimized igllama startup.

igllama startup — optimized

# 0.8B Q8 — 764MB loaded, ~3–5s/call
igllama api Qwen3.5-0.8B-Q8_0.gguf \
  --port 8090 --no-think --max-tokens 512 \
  --threads 4 --threads-batch 16 --ctx-size 2048 --mlock &

# 4B Q8 — 4.2GB loaded, ~10–65s/call
igllama api Qwen3.5-4B-Q8_0.gguf \
  --port 8091 --no-think --max-tokens 512 \
  --threads 4 --threads-batch 16 --ctx-size 2048 --mlock &

# 0.5B coder Q8 — 645MB loaded, ~0.5–5s/call
igllama api qwen2.5-coder-0.5b-instruct-q8_0.gguf \
  --port 8092 --max-tokens 512 \
  --threads 4 --threads-batch 16 --ctx-size 2048 --mlock &

Why --threads 4? On a 16-core CPU, using all cores for generation means 16 threads competing for memory bandwidth to load model weights. Limiting to 4 generation threads reduces bandwidth contention per thread — empirically, 4 threads produces higher token/s than 16 for inference-bound workloads with Q8 models this size. --threads-batch 16 uses full parallelism only for the cheaper prompt-evaluation phase.

Qwen3.5-0.8B-Q8 (5/13): The 0.8B model’s json_mode output consistently double-escapes internal quotes: {"args\":{"command":"..."}} instead of {"args":{"command":"..."}}. The brace-counter extractor can’t recover these because the structural quotes are escaped. When the prompt contains no quoted substrings in the expected bash command, the 0.8B occasionally produces valid JSON — hence 5/13. This is a model training gap, not an igllama issue; the 0.8B was not trained to produce structured JSON with complex nested string values.

Qwen2.5-0.5B-coder-Q8 (1/13): The 0.5B coder model doesn’t engage with the tool loop at all. It answers questions directly in plain text or wraps responses in markdown code blocks. json_mode produces {"answer": "..."} responses — correct JSON structure, but always choosing answer as the key regardless of what tool is needed. The one “pass” was T06 (struct names), which hit the retry limit and returned an HTTP 400 error that was spuriously counted as done.

New findings from extended trial

  1. The timeout fix works. --threads 4 --max-tokens 512 --ctx-size 2048 --mlock reduced 4B Q8 per-call latency from 120s+ (timeout) to 10–65s. T03 passed immediately.
  2. 0.8B json_mode double-escape is structural. The 0.8B model escapes quotes inside its bash command strings, producing malformed JSON. This is not recoverable by the extractor — it requires either a different prompt format (colon-separated, no JSON) or a model fine-tuned on function-call JSON.
  3. 4B fails on self-referential grep. T02 (counting enum variants) failed because the model narrowed its grep pattern too aggressively (grep 'idle,') and couldn’t recover when only one variant appeared in results. The model needed to read the enum block directly (grep -A 20 'LoopState = enum'). This is a grepping strategy failure, not a JSON or tool-call failure.
  4. 4B hallucinates missing files. In T11, the model fabricated src/agent/registry.zig — a file that doesn’t exist. When head -n 5 registry.zig returned exit 1, the model issued ls src/agent/ to recover, but ran out of turns before completing all four files.
  5. Shell arithmetic beats model arithmetic. In T12, the 4B model correctly used echo $((503 * 2)) rather than computing 1006 in its generation — the right call for verifiable, exact arithmetic.

Engineering requirements (updated)

Five requirements for reliable multi-turn tool use with a local model:

  1. JSON-mode — response_format: {"type":"json_object"} forces structured output; without it the model writes markdown prose
  2. Brace-balanced JSON extractor — json.loads() fails on raw responses ~30% of the time; brace-counting tolerates trailing garbage
  3. Retry with backoff on 400 — igllama returns spurious 400s under load; 3 retries × 5s absorbs them
  4. Context window capping — keeping only the last 4 tool/result pairs prevents context overflow (which also produces 400s)
  5. igllama thread tuning — --threads 4 --threads-batch 16 --mlock on a 16-core CPU; more threads ≠ faster inference for memory-bandwidth-bound Q8 models

Dogfooding Verdict

The Ralph Loop is validated at every scale tested. The explicit state machine keeps the agent on track whether the model is 4B running locally or Claude Opus running in the cloud — it drives the model through states, it doesn’t rely on the model to self-sequence.

For multi-turn coding tasks with a local 4B model, the requirements are concrete:

  • igllama thread tuning (--threads 4 --threads-batch 16 --mlock) to prevent memory-bandwidth saturation
  • json_mode to force structured output
  • A brace-tolerant JSON parser to handle igllama’s trailing-byte edge cases
  • Retry logic for server throttling
  • Context trimming to stay inside the inference window

With those five things in place, Qwen3.5-4B at Q8_0 quantization completes 10 of 13 task types — including multi-step read → write → verify workflows, self-correction from tool errors, shell arithmetic chains, and 5-turn grep-to-file pipelines. The three failures are specific grepping strategy gaps (T02), markdown regression under a long context (T08), and hallucinated file names (T11) — not fundamental tool-call or JSON formatting problems.

That is the full agentic coding loop at 4B scale. Slower, free, and fully local.
